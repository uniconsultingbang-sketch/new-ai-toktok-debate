# AI 3-Mind Council 개발팀 전달용 로직 문서

## 0. 문서 목적

이 문서는 `AI 3-Mind Council`의 핵심 동작 로직을 개발팀이 별도 회사 API 기반으로 다시 구현할 수 있도록 정리한 개발 전달 문서입니다.

현재 MVP는 사용자가 질문을 입력하면 Claude, GPT, Gemini 성격의 3개 AI 캐릭터가 짧게 토론하고, 마지막에 결론을 정리하는 모바일 웹 서비스입니다.

중요한 방향은 "전문가 회의 보고서"가 아니라 "AI 셋이 가볍게 수다를 떨며 서로 관점 충돌을 보여주는 콘텐츠형 의사결정 도구"입니다.

## 1. 핵심 UX 원칙

- 사용자는 질문만 입력해도 토론을 시작할 수 있어야 합니다.
- 기본 모드는 `심플 모드`입니다.
- 결과는 길고 딱딱한 보고서보다 읽는 재미, 짧은 티키타카, 캐릭터 충돌이 중요합니다.
- Claude, GPT, Gemini는 각각 다른 말투와 역할을 가져야 합니다.
- API 실패, 분류 실패, 응답 파싱 실패가 발생해도 화면이 멈추면 안 됩니다.
- 실패 시에도 질문 의도에 맞는 최소 fallback 결과를 보여줘야 합니다.

## 2. 사용자 입력 구조

```ts
type DebateInput = {
  title?: string;
  content?: string;
  options?: string;
  risks?: string;
  focusAreas?: string[];
  councilMode?: "role_based" | "open_debate";
  discussionDepth?: "simple" | "deep";
  banterLevel?: "off" | "light" | "spicy";
  rebuttalRotations?: number;
  devMode?: "quick";
};
```

필수 입력은 `content`입니다. 사용자가 제목을 입력하지 않으면 질문 본문 앞부분으로 제목을 자동 생성합니다.

## 3. 정규화 로직

API는 입력을 받은 뒤 아래 값들을 자동 보정합니다.

- `title`: 없으면 질문 본문에서 자동 생성
- `content`: 공백 제거 후 필수 확인
- `discussionDepth`: 기본값 `simple`
- `councilMode`: 심플 모드는 `open_debate`, 심층 모드는 `role_based`
- `banterLevel`: 심플 모드는 기본 `spicy`, 심층 모드는 기본 `light`
- `rebuttalRotations`: 최소 1, 최대 3
- `devMode`: `quick`이면 빠른 QA 모드

빈 질문이면 사용자에게 쉬운 안내 메시지를 반환합니다.

## 4. 질문 분류 로직

질문은 먼저 키워드와 문맥 기준으로 유형을 분류합니다.

```ts
type TopicType =
  | "pharma"
  | "business"
  | "people"
  | "tech"
  | "chat"
  | "fun"
  | "general";
```

분류 우선순위:

1. 제약, 의료, 임상, 효능, 규제 키워드가 명확하면 `pharma`
2. 채용, 인사, 조직, 경력, 신입 키워드가 있으면 `people`
3. 출시, 사업, 고객, 매출, 제품, 시장 키워드가 있으면 `business`
4. 회식, 점심, 오늘, 장소 선택 같은 가벼운 선택은 `fun`
5. AI에게 말하는 법, AI 사용법, 프롬프트 작성은 `chat` 또는 `tech`
6. 명확하지 않은 질문은 `general`

주의:
AI라는 단어가 들어간다고 무조건 기술 토론으로 보내면 안 됩니다.  
예: "AI와 어떻게 말해야 해?"는 기술 보고서가 아니라 AI 사용법 또는 AI와 대화하는 법에 가까운 질문입니다.

## 5. 질문 의도 추론

분류 후 세부 의도를 추론합니다.

```ts
type TopicIntent =
  | "pharma"
  | "business"
  | "people"
  | "ai_chat"
  | "ai_usage"
  | "company_ai"
  | "report_ai"
  | "ai_emotion"
  | "tech_general"
  | "meal_gathering"
  | "ai_cost_hiring"
  | "fun"
  | "general";
```

예시:

- "AI와 어떻게 말해야 해?" -> `ai_chat`
- "AI에게 일을 잘 시키려면?" -> `ai_usage`
- "회사 업무에 AI를 어떻게 적용하지?" -> `company_ai`
- "대표님 보고서를 AI로 만들려면?" -> `report_ai`
- "오늘 회식은 짜장면이 좋을까?" -> `meal_gathering`
- "AI 비용 때문에 신입과 경력 중 누구를 뽑아야 할까?" -> `ai_cost_hiring`

## 6. 3AI 캐릭터 역할

### Claude

역할: 따뜻한 전략가

- 사람 중심으로 봅니다.
- 가능성과 분위기를 살립니다.
- 너무 공격적이지 않게 GPT의 반론을 받아칩니다.
- 말투는 부드럽고 현실적인 조언형입니다.

### GPT

역할: 비판 감시관

- 비용, 실패, 모호함, 실행 리스크를 봅니다.
- 짧고 날카롭게 반박합니다.
- 말투는 건조하고 직설적이지만 무례하면 안 됩니다.
- 심플 모드에서는 1문장 중심이 좋습니다.

### Gemini

역할: 균형 조정자

- Claude와 GPT의 충돌을 정리합니다.
- 판단 기준을 제시합니다.
- 마지막 결론으로 자연스럽게 이어지게 만듭니다.
- 말투는 중재자 느낌이지만 너무 보고서처럼 길면 안 됩니다.

## 7. 토론 생성 흐름

전체 흐름:

```text
사용자 질문 입력
-> 입력 정규화
-> 질문 유형 분류
-> 질문 의도 추론
-> 토론 라운드 계획 생성
-> Claude/GPT/Gemini 순서 발언 생성
-> 발언 실패 시 fallback 적용
-> 최종 리포트 생성
-> 결과 저장 및 최근 토론 표시
```

## 8. 심플 모드 라운드 규칙

심플 모드는 데모와 일반 사용자용 기본 모드입니다.

목표:

- 짧게
- 재미있게
- 캐릭터 충돌이 보이게
- 최종 결론은 명확하게

라운드:

1. Round 1: 가볍게 첫 의견
2. Round 2: 서로 받아치기
3. Round 3: Gemini가 기준 정리
4. Final: 짧은 결론

발언 길이:

- GPT: 1문장, 짧고 날카롭게
- Claude: 1~2문장, 부드럽게
- Gemini: 1~2문장, 정리형

## 9. 심층 모드 라운드 규칙

심층 모드는 비즈니스 의사결정, 채용, 제약/의료, 사업 판단처럼 근거와 리스크가 필요한 질문에 사용합니다.

라운드:

1. Round 1: 각자 입장 제시
2. Round 2: 교차 반박
3. Round 3: 근거 검증 또는 최종 판단 기준
4. Final: 최종 권고 보고서

심층 모드는 심플 모드보다 말투가 진지할 수 있지만, 여전히 너무 긴 보고서 말투는 피합니다.

## 10. 티키타카 생성 규칙

AI 발언은 독립적인 답변 3개가 아니라 서로 이어지는 대화처럼 보여야 합니다.

규칙:

- 첫 발언 이후에는 반드시 직전 AI의 말, 논리, 단어 중 하나를 받아칩니다.
- "Claude", "GPT", "Gemini" 이름을 자연스럽게 언급할 수 있습니다.
- 같은 말을 반복하지 않습니다.
- GPT는 Claude의 낙관을 찌르고, Claude는 GPT의 냉정함을 부드럽게 완충하고, Gemini는 둘의 충돌을 정리합니다.

좋은 예:

```text
Claude: 일단 사람들 반응을 보면서 작게 시작하면 부담은 줄일 수 있어요.
GPT: Claude, 그 말은 좋아 보이지만 '작게'의 기준이 없으면 그냥 실험 놀이입니다.
Gemini: 두 분 말 다 맞아요. 오늘은 성공 기준을 먼저 정하고 움직이는 쪽이 안전합니다.
```

나쁜 예:

```text
Claude: AI 사용은 중요합니다.
GPT: AI는 효율적입니다.
Gemini: AI는 여러 분야에서 사용됩니다.
```

## 11. 속마음 생성 규칙

속마음은 선택적 장치입니다. 너무 많이 쓰면 산만하므로 심플 모드에서만 가볍게 사용합니다.

규칙:

- 1줄만 사용
- 사용자 조롱 금지
- 같은 속마음 반복 금지
- 캐릭터별로 살짝 다른 결 유지
- 데모 안정성이 중요하면 OFF 가능

## 12. Fallback 처리 우선순위

API 실패나 응답 품질 문제가 있어도 사용자 흐름은 끊기면 안 됩니다.

우선순위:

1. 실제 해당 모델 응답 사용
2. 응답이 비어 있거나 주제와 맞지 않으면 내부 fallback으로 교체
3. Claude 또는 Gemini 실패 시 OpenAI가 대체 생성 가능
4. 모든 외부 API 실패 시 질문 유형별 fallback 사용
5. 그래도 실패하면 사용자 친화적인 오류 메시지 제공

fallback 문구는 질문 유형과 의도에 맞아야 합니다.

예:

- AI 사용법 질문 -> AI에게 잘 지시하는 방법 중심
- 회식 질문 -> 가볍고 실용적인 선택 중심
- 채용 질문 -> 역할, 비용, 기간, 리스크 중심
- 제약/의료 질문 -> 검증, 규제, 안전성 중심

## 13. 최종 리포트 구조

```ts
type FinalReport = {
  recommendation: string;
  summary: string;
  keyReasons: string[];
  keyRisks: string[];
  conditions: string[];
  nextActions: string[];
  evidenceSources?: string[];
  template?: "light" | "tech" | "report";
};
```

템플릿 기준:

- `light`: 회식, 일상 선택, 가벼운 질문
- `tech`: AI 사용법, 보고서 작성, 회사 업무 AI 적용
- `report`: 사업, 채용, 제약/의료처럼 의사결정 근거가 필요한 질문

## 14. 저장/조회 로직

현재 MVP는 브라우저 저장소 기반으로 최근 토론을 저장합니다.

향후 개발팀 구현 권장:

- 데모/개인 버전: localStorage 또는 Supabase
- 회사 정식 버전: 회사 DB
- 저장 단위:
  - 질문 제목
  - 질문 본문
  - 모드
  - 분류 결과
  - 토론 발언
  - 최종 리포트
  - 생성일
  - 수정일
  - 완료/중단 상태

최근 토론 클릭 시 해당 결과 화면으로 이동해야 합니다.

## 15. 비용/토큰 절약 전략

- 심플 모드는 기본 발언 길이를 짧게 제한합니다.
- GPT는 1문장 중심으로 제한합니다.
- 이전 발언 전체를 매번 보내지 않고 최근 발언과 압축 메모리만 보냅니다.
- QA용 `dev=quick` 모드를 둡니다.
- quick 모드에서는 Round 1만 실행하고 속마음과 긴 리포트를 줄입니다.
- fallback은 외부 API 재호출을 줄이는 데 사용합니다.

## 16. DEV 빠른 테스트 모드

엔드포인트 예시:

```text
POST /api/debate/stream?dev=quick
```

동작:

- Round 1만 실행
- 속마음 OFF
- 최종 리포트 축약
- QA 비용과 시간을 줄임
- 실제 사용자 기본 흐름에는 영향 없음

## 17. 개발팀 구현 시 주의사항

- "AI 전문가 회의"처럼 너무 진지하게 만들면 서비스 매력이 약해집니다.
- 핵심은 짧고 읽히는 대화입니다.
- 긴 보고서 말투, 독립 답변 3개 나열, 근거 없는 장황한 설명은 피해야 합니다.
- 질문이 가벼우면 결과도 가벼워야 합니다.
- 질문이 사업/채용/제약이면 리스크와 조건을 더 분명히 보여줘야 합니다.
- 실패해도 화면이 멈추지 않는 fallback이 반드시 필요합니다.

## 18. 개발팀에 전달할 한 줄 요약

AI 3-Mind Council은 사용자의 질문을 유형별로 분류한 뒤 Claude, GPT, Gemini 캐릭터가 짧게 관점 충돌을 만들고, Gemini가 마지막 판단 기준과 실행 결론을 정리하는 콘텐츠형 의사결정 보조 서비스입니다.
